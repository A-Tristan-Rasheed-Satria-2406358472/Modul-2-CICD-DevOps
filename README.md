# Refleksi 2

1. Setelah menulis unit test saya merasa lebih yaking (seharusnya) terhadap bagian yang sudah diuji karena regresi bakal jadi lebih mudah terdeteksi. Unit test fokusnya pada cakupan perilak dimana setiap alur yang penting harus punya test (positif, negatif, dan kasus batas). Untuk memastikan testnya cukup, bisa dengan gabungkan unit tests dengan integration/functional tests, pikirkan tentang skenario edge-case, dan gunakan metrik seperti code coverage untuk menemukan area yang belum teruji. menurut saya 100% code coverage belum berarti berarti kode bebas bug, coverage cuman memperlihatkan baris yang dieksekusi oleh test, bukan kualitas atau kecakupan semua kondisi logika

2. Jika membuat kelas functional test baru yang menyalin setup dan instance variable dari test lain, ini berpotensi menurunkan kualitas kode karena duplikasi, jadinya akan lebih susah maintenance, dan risiko ketidakkonsistenan ketika setup berubah. Untuk memperbaikinya bisa dengan ekstrak konfigurasi dan inisialisasi ke superclass (ex : BaseFunctionalTest) atau helper/fixture bersama. menggunakan @BeforeEach untuk inisialisasi yang konsisten.menggunakan juga Page Object Pattern untuk functional tests agar interaksi UI terenkapsulasi. menggunakan parameterized tests bila hanya input yang berubah. Langkah-langkah ini bisa mengurangi duplikasi, mempermudah perubahan, dan meningkatkan keterbacaan serta maintainability test suitenya
